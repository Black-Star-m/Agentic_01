
## 一、智能体工作流
### 构建智能体
构建智能体工作流的关键之一：
	分析别人做的事情，并识别可以用哪些独立步骤来实现；
	当分析每一个独立步骤是，思考这个步骤是否能用大语言模型LLM实现，或者能用访问的API或函数调用来实现；如果不能，是否可以进一步拆解这个步骤，把它分解成更小的步骤，这样或许更适合用大语言模型或软件工具实现。

构建智能体工作流时，先做一个初步的任务拆解，也就是初步的智能体工作流，然后需要不断迭代和优化流程，直达达到期望的性能水平。为推动这个改进过程，其中一个关键就是评估智能体工作流。评估方法是构建并持续优化的工作流以获得理想性能的关键环节

OpenClaw ≈ 一个内置了「严肃编程式流程 + 可扩展算子系统」的 Agent 框架
OpenClaw 就是 Agent：标准感知-规划-工具-执行闭环。
符合严肃编程思路：强流程、结构化推理、Skill=算子、可优化可扩展。
​是成品 Agent 框架：开箱即用，自带聊天入口/记忆/工具，可本地部署。
​支持集群化：主从多智能体协作，能直接扩成你要的「智能体集群工具系统」

#### 智能体AI评估方法
1. 端到端评估，衡量整个智能体的输出质量；端测（端到端测试，整个智能体全链路流程）
2. 组件级评估，评估智能体流程中某一步的输出质量；单测（单元测试，单个智能体组件、单个函数；局部部分）
助于推动开发流程的不同环节；检查中间输出--大模型的轨迹，这样可以了解哪些方面未达到预期，即错误分析；就是逐步阅读每个中间输出，寻找改进机会。

### 智能体设计模式（思考如何构建模块组合成更复杂的工作流）
1. 反思-AI生成结果后，自己检查自己的输出，发现问题再修改；
2. 工具使用-AI能调用外部工具，比如搜索网页、运行代码、查数据库；
3. 规划-面对复杂任务，AI先拆解步骤、制定计划，在一步步执行；
4. 多智能体协作-多个AI智能体分工合作，各有专长，相互配合完成任务；

## 二、反思

反思是大模型/智能体的自我校验、自我复盘、自我修正的设计模式。
流程：初次生成输出 → 自我检查、批判、找漏洞 → 修正原有答案/方案 → 输出优化后结果

核心特点：
	
- 反思模式不能让LLM每次都百分百正确，但通常能带来一定程度的提升，能稳定提升输出质量、逻辑性、准确率；
- 需注意的是，当有新的外部信息时（将大模型每次验证测试的输出或者错误信息作为额外输出；联想Transformer里的额外输出），可以将这些新信息纳入反射流程进行处理，LLM能更深入地反思，并找出可能的错误原因。反射会更为强大；

#### 反思和直接生成的区别

1. 直接生成（普通LLM模式）想到什么输出什么；
- 逻辑：用户提问 → 模型一次性直接生成答案
- 特点：单次推理、没有回头检查；错了就错了，不会自查
- 速度快，但容易逻辑漏洞、事实错误、细节遗漏
- 定位：单次正向推理，一锤子买卖
 
1. 反思生成（Agent反思模式）先写完，自己检查一遍，改完再交卷。
- 逻辑：先生成答案 → 自己当“考官”审自己 → 挑错、找问题 → 重新优化生成
- 特点：多轮推理，正向生成 + 反向自检；主动发现逻辑矛盾、计算错误、信息缺失
- 速度稍慢，但严谨度大幅上升
- 定位：生成 + 校验 + 迭代优化 的闭环

#### 评估反思的影响
1. 正确率/准确率
对比「直接生成」vs「带反思生成」的任务正确率，反思后一般有明显上涨。
2. 错误类型减少程度
逻辑错误、步骤遗漏、前后矛盾、常识错误，这类问题会显著减少。
3. 任务完成完整度
复杂任务下，反思能补全遗漏步骤，让工作流更完整、闭环更通顺。
4. 副作用
推理轮次变多、Token消耗增加、响应时延变长（成本换质量）。

#### 外部额外信息提升反思效果（重点）

	反思 + 外部真实反馈/报错信息 = 智能体自我进化的核心闭环
	没有外部信息，反思是“瞎自查”；
	有了外部信息，反思是“对着标准答案改错题”，能力直接质变。

1. 无外部信息的反思
只有模型自身知识+自我主观判断来复盘
- 只能靠自己“脑补”哪里错
- 反思能力有限，提升幅度一般
 
1. 有外部额外信息加持的反思（效果暴涨）
把这些作为反思的输入素材：
- 程序运行报错日志
- 单元测试/端到端测试的失败结果
- 工具调用返回的原始数据
- 外部检索到的最新事实信息
- 上一轮输出的错误反馈
 
类比 Transformer 思想：
把测试报错、运行日志、错误反馈看成模型的额外监督信号、额外输出信息，一并送入反思流程。
 
逻辑链条
1. 智能体第一次生成方案/代码/回答
2. 执行后得到错误信息、测试结果、外部真实数据
3. 将这些外部信息全部喂给 LLM 做二次反思
4. 模型根据真实报错，精准定位错误原因
5. 修正后再次生成，下一轮即使没有外部信息，效果也会更好

## 三、工具

 Agent中工具是不是又分为固定的工具(Static Tools)，类似硬编码，或者系统提示词那样，属于框架内置能力，LLM 只能选择用或不用，不能自己创造新工具；
 同样的还有不固定的工具(Dynamic Tools)，LLM会根据实际情况决定是否使用，类似软编码，用户提示词那样，LLM自主调用相关函数，保存到记忆系统内，根据上下文内容来推断使用那个函数/工具

0. 系统Prompt告诉LLM有哪些些工具，如CALCULATOR、SEARCH，格式必须大写，怎么传参数，这里不涉及工具的具体实现，只是相当于一个工具说明文档；
1. LLM会根据用户的提示词推断出是否需要使用工具；如果用户的提示词中表示需要使用工具，LLM会理解到，并输出一个工具调用文本，这个工具调用文本中的工具名字是大写的；
接着就有了反思的那套逻辑，LLM可能无法从用户提问到调用工具一次性实现，因此要分两次，也就是第二轮调用工具接口时，要再次“回调”一次。
2. LLM会根据这个输出的调用文本，联系上下文，再次回调一次，去调用相关的工具（注意这里LLM自己不实现工具，只调用已有的工具接口），工具根据LLM的输出执行，然后得出一个结果再次给LLM；
3. LLM会再次根据这个输出的结果转化成人类语言返回给用户，整个过程其实就是反思的一个执行流程；

代码执行：沙盒环境（Docker、E2B）

MCP是LLM获取更多上下文和工具的方式

## 四、评估
#### 用评估驱动开发流程
- 构建系统，通过查看输出结果定位变现不佳的环节；先上线，跑通基础流程，再通过真实输出找问题；
- 用小评估闭环迭代，通过搭建一个包含约20个样本的小型评估，避免大规模标注的成本，
- 在修改工作流（例如新提示词、新算法）时进行监控，观察指标是否提升，追踪每一次修改的效果；
- 查看输出，发现问题，如果知道怎么修改，就直接修改；但如果需要更长的改进过程，就建立评估集，用它推动后续开发；
智能体系统通常包含许多部分，清楚花时间改进哪一部分最有成效，能够做好这一点对于推动高效发展来说是一项非常重要的技能；

#### 错误分析
在完成智能代理系统整体架构搭建后，需对业务工作流中的各个执行环节进行全流程监测与逐项校验，以此判断每一步流程是否能够正常调度与执行。系统运行过程中，若某一功能组件或流程步骤频繁产生异常与执行错误，该类问题节点将作为系统优化的关键依据。
通过统计各模块、各步骤的错误发生频次与故障影响范围，可以明确问题修复的优先级，将调试与优化资源集中在高报错、高风险的流程节点上。该错误分析机制能够快速定位系统薄弱环节，避免无差别盲目排查，有效提升智能代理系统的故障排查效率与后续迭代优化效果。
利用误差分析来确定重点关注领域，对提高效益非常有益。

#### 组件级评估
组件级评估，就是不把整个智能体系统当成一个整体去打分，而是把系统拆成一个个独立小组件/小模块，单独对每个模块做性能、效果、稳定性评估。
1. 先拆分系统：比如规划模块、工具调用模块、记忆模块、推理模块、执行模块等
2. 逐个单独测：每个组件单独跑任务，看它会不会出错、准确率怎么样、响应稳不稳定
3. 目的：
- 快速知道哪个组件拖后腿
- 不用整体排查，直接定位坏的模块
- 知道该优先优化哪一部分
把大系统拆成各个零件，逐个零件单独评测好坏，就是组件级评估。

#### 解决方法
Agentic workflow 的整体构成
- 智能体工作流由多种不同组件构成，不同组件的优化方法差异很大。
- 整体分为两大类：非 LLM 组件、LLM 组件。

非 LLM 组件包括：
- 网页搜索引擎
- RAG 文本检索模块
- 代码执行模块
- 独立训练的ML模型（语音识别、图像人物检测等）
优化手段主要两类：
1. 调参 / 调超参数- 搜索：结果数量、时间范围
- RAG：相似度阈值、文本分块大小（chunk size）
- 目标检测：置信度阈值（平衡误报/漏检）
强调：具体参数不重要，重要的是这类组件通常都有可调节参数。
2. 直接替换组件- 更换不同的 RAG 引擎、检索服务商
- 对比测试哪个效果更好
因为非 LLM 组件种类多，优化方法高度依赖具体功能，没有统一套路。
 
LLM 组件的优化方法（从简单到复杂）
按成本 & 难度从低到高排序：
1. 优化提示词（Prompt）- 加入更明确、具体的指令
- 使用小样本学习（few-shot prompting），给输入输出示例，让模型学得更快
2. 更换不同 LLM 模型- 用 AI Suite 等工具快速切换多个模型
- 通过评测（evals）选出最适合任务的模型
3. 复杂任务拆解- 单步太复杂 → 拆成多步
- 例如：先生成 → 再反思（reflection）
- 连续调用 2～3 次简单步骤，比一次复杂指令更容易准确执行
4. 模型微调（Fine-tuning）- 复杂度最高、开发成本最高
- 效果通常显著优于仅靠提示词
- 作者习惯：其他方法全部用尽后才微调
- 适用场景：已达到 90%～95% 效果，想榨干最后几个百分点
- 多用于成熟、稳定上线的项目
 
如何培养对 LLM 模型能力的直觉（重点举例）
不同模型能力差异明显：
- 大模型（前沿模型）：更擅长遵循复杂指令
- 小模型（如 Llama 3.1 8B）：适合简单的事实性问题，复杂指令容易跑偏，不太擅长遵循指示；
举例：PII （个人身份信息）隐私信息脱敏任务
- 小模型：容易漏检姓名、地址，输出格式混乱，不严格按指令来
- 能力更强的模型：能完整识别并正确脱敏所有 PII，严格遵循输出格式
不同模型各有所长：有的擅长代码，有的擅长指令遵循，有的擅长特定领域知识；
建立直觉后，选型和写 prompt 都会更高效。

提升模型直觉的具体做法
1. 多试用新模型- 闭源、开源都试
- 用各种查询测试能力边界
2. 建立自己的评测集（personal evals）- 固定一批测试用例，横向对比不同模型
3. 大量阅读别人的 Prompt- 看网上公开的 prompt
- 和业内朋友交流 prompt 写法
- 扒开源项目里作者写的 prompt 学习最佳实践
4. 在实际 agentic workflow 中反复测试- 看链路执行痕迹（traces）
- 做组件级 / 端到端评测
- 同时感受效果、速度、成本三者权衡
提升各组件性能 → 提升端到端系统质量

#### 先质量，后成本与延迟
优先保证输出质量，成本和延迟后期再优化，输出质量是最难做好的部分，先把效果跑通；
成本和延迟重要，但优先级更低
 
- 工作流上线后用户暴增，成本才变成问题，再紧急优化，这属于“幸福的麻烦”，所以前期不必过度担心成本；
 
 延迟（latency）优化方法：
1. 对每个步骤计时、做基准测试（benchmark）
例如：
- LLM 生成搜索词：7s
- 网页搜索：5s
- 最终文章生成：18s
2. 找到耗时最长的环节重点优化
3. 可行手段：
- 对可并行步骤（如网页抓取）实行并行执行
- 耗时过长的 LLM 步骤，尝试更小、更快的模型
- 更换更快的 LLM 服务商（有些有专用加速硬件）
 
 成本（cost）优化方法：
1. 计算每一步的成本
- LLM：按输入输出 token 计费
- 搜索/工具：按 API 调用次数计费
- 计算服务：按服务器资源计费
2. 找到成本最高的步骤重点优化
3. 可行手段：
- 替换更便宜的组件 / LLM
- 精简 token 消耗
4. 基准测试能清晰看出：哪些步骤成本占比极低，不值得优化
 
优化成本和延迟的核心方法就是：先测量，再针对性优化；
- 测量每一步的延迟和成本 → 定位瓶颈 → 集中优化
​
#### 构建 Agentic 系统的真实迭代流程（收尾）
构建智能体系统不是线性开发，而是在「构建」和「分析」之间反复循环。
两大核心活动
1. Building（构建）：写代码、搭系统、调组件、实现端到端流程。
2. Analysis（分析）：看似没“产出”，但同等重要，决定下一步优化方向。
 
完整迭代阶段（从粗糙到严谨）
1. 快速搭出端到端系统
先做一个简陋可用版本，跑通整体流程。
2. 人工看结果 + 看 traces
观察哪里好、哪里崩，凭直觉定位问题组件。
3. 建立简单评测集（10～20 条样本）
计算端到端指标，更系统地判断系统表现。
4. 做错误分析（error analysis）
统计每个组件导致劣质输出的频率，精准定位瓶颈。
5. 建立组件级评测（component-level evals）
对成熟系统做精细化评估，高效迭代单个模块。
 
- 高度循环往复：调整体 → 查错误 → 改组件 → 评测 → 再调整体
- 新手常见问题：只疯狂写代码，很少做分析、错误分析、搭建评测，效率极低
- 分析能让你把时间花在真正值得改的地方
 
工具与评测使用建议
- 有很多工具可做追踪、日志、耗时、成本监控
- 但实际项目中的 Agentic Workflow 往往高度定制
- 因此通常要会自己写定制化评测（custom evals），更贴合具体问题
- 通用工具辅助 + 定制评测为主

## 五、高度自主智能体（规划）
##### 智能体规划设计模式（Planning Design Pattern）
用于构建高度自主的智能体；无需提前硬编码*任务执行步骤，让智能体自主灵活决策步骤顺序*，完成复杂任务。
***案例1：太阳镜零售门店客服智能体***
1. 业务场景
门店库存存在数据库中，需智能体回答复杂客户问题，例如：有没有在售的圆形太阳镜，且价格低于100美元？
该问题需三步筛选：查款式（圆形）→查库存（在售）→查价格（＜$100）。
2. 给LLM配备的工具集
为让LLM处理各类查询，提供一系列工具（部分工具仅适配其他场景）：
- 获取商品描述（筛选圆形太阳镜）
- 检查库存
- 处理退货
- 获取商品价格
- 查询历史交易
- 处理商品销售
3. 提示词设计思路
向LLM输入提示词：
- 告知可调用的所有工具及功能描述
- 要求LLM输出分步执行计划，完成用户请求
4. LLM生成的执行计划
针对该查询，LLM会输出标准三步计划：
- 调用「获取商品描述」，筛选出圆形太阳镜
- 调用「检查库存」，确认圆形款是否有货
- 调用「获取商品价格」，筛选库存中价格＜$100的商品
5. 计划执行流程（核心执行逻辑）
- 将步骤1指令+任务背景+工具说明输入LLM，执行第一步，得到圆形太阳镜清单
- 将步骤1结果+步骤2指令输入LLM，执行第二步，得到有货的圆形太阳镜
- 将步骤2结果+步骤3指令输入LLM，执行第三步，得到符合价格要求的商品
- 最终将所有步骤结果输入LLM，生成给客户的自然语言答复
6. 细节补充
- 实际LLM生成的计划比演示的单行指令更详细
- 核心优势：无需开发者预设工具调用顺序，不同客户问题可生成不同计划
- 扩展场景：客户提出“退金框眼镜、保留金属框眼镜”，LLM可自主生成新计划：查历史交易→筛选金框眼镜→调用处理退货工具
 
***案例2：邮件助手智能体***
1. 任务需求
指令：回复Bob来自纽约的邮件邀请，告知会参加，并归档该邮件。
2. 配备工具
搜索邮件、移动邮件、删除邮件、发送邮件。
3. LLM生成的执行计划
- 调用「搜索邮件」，找到Bob提及纽约/晚餐的邀请邮件
- 生成并调用「发送邮件」，回复确认参加
- 调用「移动邮件」，将该邮件归档
4. 执行逻辑
与太阳镜案例一致：分步执行，每一步将上一步结果+当前步骤指令输入LLM，依次完成。
 
 规划设计模式让LLM自主规划复杂软件的开发步骤（拆分组件、分步构建），按计划逐步完成复杂软件开发。同时系统可控性变差，开发者在运行时无法预知LLM生成的具体计划，难以提前管控执行逻辑。
 - 系统提示词会告诉LLM可调用的工具，要求以JSON格式生成分布规划
 - LLM生成的规划是JSON格式；JSON格式结构清晰、无歧意，下游代码可以稳定解析步骤、工具、参数，目前主流大模型都擅长生成规范JSON；
 - JSON规划整体是一个步骤列表，每一步包含明确的键值对（步骤描述，要调用的工具，传给工具的参数），代码可逐步读取并依次执行；

#### 规划并执行（通过让LLM写代码来执行LLM输出的JSON规划）
用代码执行实现规划（Code-based Planning）
- **无需让LLM输出JSON等结构化规划再分步执行，而是直接让LLM编写代码，用代码承载规划的多步逻辑**，再由代码执行器（如Python解释器）运行代码，完成复杂任务；
- LLM既当架构师（做规划），又当程序员（写执行代码），是更高效更强大的Agent规划方式。
核心优势：代码天然支持循环、条件分支、异常处理等复杂逻辑，能解决JSON规划无法覆盖的场景，且无需开发者不断新增工具。

场景：基于咖啡销售数据表，回答用户各类查询（如“哪个月热巧克力销量最高”“上周有多少笔唯一交易”）。
传统方案：给LLM配备基础工具（获取列最大值、平均值、筛选行、求和等），但存在明显问题：
- 复杂查询操作繁琐：如查询“哪个月热巧克力销量最高”，需逐月份筛选、统计，再取最大值，步骤冗余；
- 工具扩展性差：遇到新查询（如“上周唯一交易数”，“最近5笔交易金额”），需不断新增工具（如“获取唯一条目”）；
- 系统脆弱、低效：开发者需持续处理边缘案例，不断补充工具，无法覆盖所有查询场景。

优化方案：提示LLM直接编写Python代码（用特定标签界定），通过代码表达规划逻辑，依托Python库（如pandas）完成数据处理。
核心逻辑：
- LLM的“架构师”角色：思考查询所需的多步规划（如加载数据→解析日期→筛选数据→统计结果）；
- LLM的“程序员”角色：将规划步骤转化为可执行代码，调用pandas等库的内置函数（无需开发者新增工具）；
- 执行主体：LLM不亲自运行代码，由代码执行器（如Python解释器）运行生成的代码，输出最终结果。
#### 具体案例演示
案例1：查询“最近5笔交易金额”
- LLM规划（代码形式）：加载CSV文件→解析日期列→按日期排序→选择最后5笔交易→提取金额列；
案例2：查询“上周有多少笔唯一交易”
- LLM规划（代码形式）：加载CSV→解析日期→定义上周时间窗口→筛选对应行→删除重复交易→统计数量。
代码中会包含注释，清晰体现LLM的规划步骤，相当于“规划+执行代码”一次完成，比JSON规划更高效。
从研究数据来看，规划效果排序：**代码规划（Code as Action）＞JSON结构化规划＞纯文本规划**
- 原因：代码可调用海量内置函数（如pandas的数百个函数），LLM熟悉这些函数的使用场景，能灵活组合完成复杂任务，无需开发者额外新增工具。

#### 代码规划的注意事项
安全风险：需使用沙箱（安全执行环境）运行LLM生成的代码，避免恶意代码或错误代码影响系统；（实际中部分开发者未使用沙箱，但不推荐）。

#### 代码规划的适用场景
并非所有场景都适用，核心适配：**可通过代码完成的任务**（如数据处理、编程开发、复杂逻辑计算等）。
其中，**高度自主的软件编码**是目前代码规划最成熟的应用场景：LLM会先规划软件组件拆分、开发步骤，再逐一步骤编写代码、测试，高效完成复杂软件开发。
#### 两种核心规划模式对比
- JSON规划（Plan & Execute）：LLM只当“架构师”，输出结构化步骤，由开发者的代码解析执行；适合简单工具调用、客服等场景，可控性强，但逻辑表达弱。
- 代码规划（Code Execution）：LLM既当“架构师”（做规划），又当“程序员”（写代码），由代码执行器运行代码；适合复杂逻辑、数据处理、编程等场景，表达能力强，但可控性弱、有安全风险。
#### 规划模式的优缺点
- 优点：无需开发者硬编码步骤顺序，大幅提升Agent的任务覆盖范围和灵活性；
- 缺点：可控性差，开发者无法提前预知LLM在运行时会生成怎样的规划（代码或JSON），难以提前管控。

## 六、多智能体系统（Multi-Agent System）
- **不再是单一Agent完成任务，而是由多个Agent协作，共同完成复杂任务**（类比人类团队分工协作）。
为什么需要多个Agent？明明可以用同一个LLM反复提示，或用一台计算机完成。
- 计算机层面：即使是单CPU，也会将工作分解为多个进程/线程，方便开发者编写代码、提升效率；
- 人类工作层面：复杂任务不会只雇一个人完成，而是组建团队，让不同角色的人负责不同子任务（如营销、科研、法律案例）。
核心价值：将复杂任务分解为多个子任务，让不同“角色”的Agent负责各自擅长的部分，开发者可专注于单个Agent的搭建，还能实现Agent复用，提升开发效率和任务完成质量。
#### 多智能体的典型应用案例
核心逻辑：复杂任务天然可分解为不同子任务，对应不同“角色”的Agent，每个Agent具备对应技能和工具。
- 案例1：创建太阳镜营销手册（核心分工）
    - 研究员Agent：分析太阳镜市场趋势、竞争对手情况，需配备“网页搜索”工具（类比人类研究员查资料）；
    - 平面设计师Agent：制作太阳镜可视化图表、宣传素材，需配备“图像生成/编辑API”，“代码执行（生成图表）”工具；
    - 撰稿人Agent：整合研究报告和设计素材，撰写营销手册文案，无需额外工具（仅靠LLM文本生成能力）。
- 案例2：撰写研究文章
    - 研究员（在线调研）、统计员（计算数据）、主撰稿人（撰写初稿）、编辑（打磨优化），分工协作完成。
- 案例3：准备法律案件
    - 律师助理、律师、调查员等不同角色Agent，各自负责对应子任务，协同推进案件准备。
#### 单个Agent的搭建方法
通过提示词让LLM“扮演特定角色”，明确其任务和所需工具，即可搭建单个Agent：
- 示例（研究员Agent提示词）：“你是一名研究Agent，擅长分析市场趋势和竞争对手，使用网页搜索工具，为太阳镜产品分析当前市场趋势，并总结竞争对手的产品情况，生成研究报告。”
同理，通过提示词定义设计师、撰稿人等角色，即可搭建对应Agent，每个Agent专注于自己的核心任务。
#### 多智能体的两种核心协作模式（重点）
1. 线性协作模式（简单直接）-线性结构
流程：按固定顺序分工，前一个Agent完成任务后，将成果传递给下一个Agent，依次推进，直至完成最终任务。
- 示例（太阳镜营销手册）：研究员Agent生成研究报告 → 平面设计师Agent根据报告制作素材 → 撰稿人Agent整合素材撰写手册。
优势：逻辑清晰、易于搭建，适合流程固定的复杂任务；可拆分开发（多人分别搭建不同Agent），还能复用Agent（如平面设计师Agent可用于营销手册、社交媒体素材等）。

2. 管理者协调模式（更灵活）-树形结构
流程：新增一个“管理者Agent”（如营销经理），由管理者Agent接收总任务、制定计划，再将子任务分配给其他Agent，协调各Agent的工作进度和成果。
具体操作：
- 给管理者Agent提示词：明确其角色（如营销经理）、可支配的Agent团队（研究员、设计师、撰稿人），要求其生成分步计划并分配任务；
- 管理者Agent制定计划：先让研究员调研趋势→设计师制作素材→撰稿人撰写手册→最后自身审核优化；
- 执行流程：管理者依次触发各Agent的任务，接收各Agent的成果，推动任务闭环。
本质：相当于多了一个“统筹者”Agent，协调其他Agent协作，适合任务流程复杂、需要灵活调整的场景。
线性协作模式只能串行执行，一个一个步骤执行；然而管理者协调模式可以并行执行，多个智能体/步骤同时执行，但这样管理者协调模式也更复杂了；
#### 不同Agent之间的通信模式（Agent如何协作、传递信息）
**不同Agent之间的通信模式**（即Agent如何协作、传递信息）。
类比人类团队协作：人类团队的沟通模式复杂，设计组织结构图需明确最优沟通方式；
多智能体系统的通信模式设计同样复杂。
##### 线性通信模式（最常用·简单易实现）
对应“线性协作模式”，通信逻辑与协作流程完全一致，呈直线传递：
- 通信流程：按固定顺序传递信息，前一个Agent完成任务后，直接将成果传递给下一个Agent，依次推进，无交叉通信；
- 示例：太阳镜营销手册制作中，研究员Agent→平面设计师Agent→撰稿人Agent，研究员仅与设计师通信，设计师仅与撰稿人通信；
- 特点：逻辑简单、易于实现和管控，适合流程固定、子任务衔接紧密的场景，是目前最基础、最常用的通信模式。
##### 层级通信模式（最常用·灵活可控）
对应“管理者协调模式”，核心是“管理者Agent统筹，各执行Agent汇报”，无直接交叉通信：
- 通信流程：管理者Agent接收总任务，分配子任务给各执行Agent；各执行Agent完成任务后，将成果**统一反馈给管理者Agent**，再由管理者Agent传递给下一个执行Agent（而非执行Agent直接通信）；
- 示例：营销经理（管理者Agent）分配任务给研究员、设计师、撰稿人；研究员完成报告后反馈给经理，经理再将报告传递给设计师，设计师完成素材后反馈给经理，经理再传递给撰稿人；
- 特点：可控性强，管理者可全程把控进度、调整任务，适合任务流程较复杂、需要统一协调的场景，与线性模式并称为最常用的两种通信模式。
##### 深度层级通信模式（较复杂·较少用）
是层级模式的延伸，形成“管理者→执行Agent→子Agent”的多层级结构，部分执行Agent可自身调用子Agent，实现更精细的分工：
- 通信流程：顶层管理者Agent分配任务给一级执行Agent（如研究员、设计师、撰稿人）；部分一级执行Agent可调用子Agent（如研究员调用“网页调研Agent”和“事实核查Agent”），子Agent完成任务后反馈给对应一级执行Agent，再由一级执行Agent反馈给顶层管理者；
- 示例：营销经理→研究员Agent→网页调研Agent、事实核查Agent；营销经理→设计师Agent（无 sub-Agent）；营销经理→撰稿人Agent→风格撰稿Agent、引用核查Agent；
- 特点：分工更精细，可处理更复杂的任务，但系统复杂度大幅提升，实现和管控难度大，目前实际应用较少。
##### 全对全通信模式（高难度·实验性）
最灵活也最难以管控的模式，所有Agent可随时与其他任意Agent通信，无固定流程和层级：
- 通信流程：给所有Agent提示，告知其可调用其他所有Agent；任意Agent可主动向其他Agent发送信息，接收信息的Agent可自主决定何时反馈；所有Agent协作直至完成任务（如撰稿人认为内容足够，或所有Agent声明完成）；
- 特点：灵活性极高，但**不可预测性强、难以管控**，结果可能存在“混乱”；适合对结果要求不高、可接受反复尝试的场景（如生成营销手册，不满意可重新运行）；目前仅用于部分实验性项目。
#### 多智能体开发框架
目前有很多软件框架支持快速搭建多智能体系统，可简化上述通信模式的实现，便于开发者探索不同通信模式的应用。
核心决策：**不同Agent之间的通信模式**（即Agent如何协作、传递信息）。
